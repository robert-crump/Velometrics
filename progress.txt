## Issue #88 — RewardComposer v1: additive flow+stop+explore reward
Completed: 2026-06-22 | Closed: pending

### What was done

**New `RewardComposer` (pure, no I/O, deterministic)**
- `domain/service/RewardComposer.kt`: object with two public functions:
  - `composeEdgeReward(edge, weights?, context?)` — scores a `MapEdge` as the
    additive sum of weighted `flow`, `stop`, and `explore` terms.
  - `composeCorridorReward(corridor, weights?, context?)` — scores a `Corridor`
    using its `pedalReward + gravityReward` as the flow term, and
    `predictedReward` as the explore term for predicted corridors only.
- `flow` (+): `pedalFlowCount + gravityFlowCount` for edges,
  `pedalReward + gravityReward` for corridors. Pedal and gravity contribute
  equally by default.
- `stop` (−): `stopPenalty` (edges only; corridors have no per-corridor stop
  signal in v1).
- `explore` (+): `exploreExploitBalance · novelty · predictedFlowScore ·
  flowConfidence`. Novelty is 1.0 for untraversed edges and predicted
  corridors, 0.0 otherwise. Predictions below the `confidenceFloor` are
  hard-excluded (explore term = 0).
- `RewardWeights` data class: per-term weight multipliers (default 1.0 each).
- `RewardContext` data class: `exploreExploitBalance` (default 0.3) and
  `confidenceFloor` (default 0.2).
- `ComposedReward` data class: carries per-component scores and the additive
  `total`.
- Inline doc block at the composition site documents deferred future terms
  (`vista`, `surface`, `quiet`, `wind`) and rejected terms (`scenery`,
  `crowding`, `training-match`).

### Tests
New `RewardComposerTest` with 13 table-driven tests asserting reward
**ordering** (relative ranking): high-flow > low-flow, pedal = gravity,
stop penalty reduces reward, confident prediction > no prediction,
below-confidence-floor exclusion, traversed edges get zero explore,
null-safe flow/stop handling, additive total, corridor ranking, predicted
vs measured corridor explore term, corridor pedal/gravity symmetry.
`compileDebugKotlin` and `testDebugUnitTest` both pass.

## Issue #23 — Surface 'Reconnect Dropbox' prompt on persistent auth failure
Completed: 2026-06-11 | Closed: pending

### What was done

Distinguishes transient sync failures from persistent Dropbox auth failures
(revoked/invalid refresh token), since retrying the latter can never succeed
without the user reconnecting.

- `data/dropbox/DropboxSyncService.kt`: `sync()` now returns a new sealed
  `DropboxSyncResult` (`Completed`, `TransientFailure`, `NeedsReauth`) instead
  of a bare `List<ImportResult>`. The single `catch (e: DbxException)` around
  the list/download loop is classified via the new internal
  `isPersistentAuthFailure()`: `InvalidAccessTokenException` or a
  `DbxOAuthException` with `DbxOAuthError.INVALID_GRANT` (revoked refresh
  token) → `NeedsReauth`; everything else (network, rate limiting, server
  errors) → `TransientFailure` if no results were produced yet, otherwise the
  partial `Completed(results)` is still returned. The cursor is only saved for
  pages that completed successfully, so transient/auth failures leave sync
  state untouched for the next attempt. `downloadAndImport()` now rethrows
  persistent-auth `DbxException`s (instead of swallowing them as a per-file
  `ImportResult.Error`) so a token revoked mid-sync is still classified
  correctly by the outer catch.
- `data/dropbox/DropboxCredentialStore.kt`: added `setNeedsReauth()` /
  `needsReauth()` backed by the same `EncryptedSharedPreferences`, cleared
  together by the existing `clear()` on disconnect.
- `data/dropbox/DropboxAuthRepository.kt`: added `needsReauth: StateFlow<Boolean>`
  (seeded from the credential store) and `markNeedsReauth()`. Cleared on a
  successful `handleAuthResult()` (reconnect) and on `disconnect()`.
- `ui/screens/home/HomeViewModel.kt`: `syncDropbox()` now switches on
  `DropboxSyncResult` — `Completed` builds the existing snackbar message and
  triggers re-clustering as before, `TransientFailure` does nothing (silent
  retry next sync), `NeedsReauth` calls `markNeedsReauth()`. Also skips
  syncing entirely while already in the "needs reconnect" state, to avoid
  repeatedly hitting the failing token-refresh endpoint.
- `ui/screens/settings/SettingsViewModel.kt`: exposed
  `needsDropboxReauth = dropboxAuthRepository.needsReauth`.
- `ui/screens/settings/SettingsScreen.kt`: when connected and
  `needsDropboxReauth` is true, shows a warning row ("Dropbox sync needs
  reauthorization") plus a "Reconnect Dropbox" button above the existing
  "Disconnect" button. Tapping it reuses `connectDropbox()` (the same OAuth
  flow from #20); the prompt clears once `handleAuthResult()` succeeds.

## Issue #71 — ScoreGpxTrack: Elevation profile section
Completed: 2026-06-13 | Closed: pending

### What was done

Added the "Elevation profile" section to the .gpx analysis overlay (#69).

- `data/gpx/GpxParser.kt`: now captures `<ele>` text inside `trkpt`/`rtept`
  elements via new `inEle`/`inTrkPt`/`inRtePt` flags, building
  `trackElevations`/`routeElevations` lists parallel to the existing
  `trackPoints`/`routePoints`. `domain/model/GpxTrack.kt` gains an additive
  nullable `elevations: List<Double?>?` field (same size as `points`),
  populated only when at least one point has a parsed `<ele>` value;
  otherwise `null`. Existing consumers (Home import flow,
  `GpxSharedViewModel`, Map tab loading) are unaffected since the field
  defaults to `null` and is purely additive.
- New functions in `util/GpxAnalysisUtils.kt`:
  - `elevationProfile(points, elevations)`: pairs cumulative haversine
    distance (m) with each point's non-null elevation, skipping points
    with null elevation.
  - `smoothElevations(elevations, windowSize = ELEVATION_SMOOTHING_WINDOW)`:
    centered moving-average smoothing (window of 5) to reduce GPS altitude
    jitter before gain/loss calculations.
  - `elevationGainLoss(smoothed)`: sums positive/negative deltas between
    consecutive smoothed points into total gain/loss (m, rounded to Int).
  - `elevationGainPer100km(gainM, totalDistanceM)`: normalizes gain to a
    per-100km figure (Int), returning 0 for a zero-distance track.
- `ui/screens/mapview/MapViewScreen.kt`: new `ElevationProfileSection`
  composable, added between `DiscoveryScoreSection` and `PoiDensitySection`
  in `GpxAnalysisOverlay`. Renders a line chart (distance vs. smoothed
  elevation) via `Canvas`/`Path`/`Stroke` following the existing chart
  patterns (e.g. `HomeScreen`'s monthly ride chart), plus gain/loss/gain-per-
  100km stats. Shows "No elevation data in this file" when
  `gpxTrack.elevations` is null or the profile has fewer than 2 points.

### Tests
Added to `data/gpx/GpxParserTest.kt`: parsing `<ele>` per `trkpt`, `<ele>`
