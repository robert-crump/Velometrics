## Issue #62 — Map-matching: anchor-by-GPS-order pruning + undirected leaf-peel safety net
Completed: 2026-06-12 | Closed: pending

### What was done

**Anchor-by-GPS-order pruning (`MapMatcher.kt`, `Constants.kt`)**
- Replaced `dropIsolatedOutliers`/`INTERVAL_MATCH_MAX_OUTLIER_RUN_POINTS` with
  `INTERVAL_MATCH_MIN_ANCHOR_POINTS = 2`. `matchTrack` now filters
  `collapseConsecutive(snapped)` runs to those with `pointCount >=
  INTERVAL_MATCH_MIN_ANCHOR_POINTS` — the high-confidence "anchors" actually
  ridden, in GPS time order — and feeds the deduped anchor sequence straight
  into `repairGaps`. Dropped low-count runs are restored by `repairGaps` as
  zero-count bridges only when needed for connectivity; genuine forks/spurs
  stay gone. Removed the redundant second `repairGaps` pass after
  `pruneLeafEdges`.

**Undirected leaf-peel safety net (`MapMatcher.kt`)**
- `pruneLeafEdges` now computes node degree via the new `undirectedDegree`,
  which collapses each edge to an unordered `(fromNode, toNode)` pair before
  counting — an anti-parallel out-and-back pair (A->B and B->A) collapses to
  one connection, so a dead-end node touched only by such a pair is correctly
  degree-1 and the whole pair is peeled in one pass. Degree is recomputed each
  iteration of the existing fixed-point loop. Start/end-node protection is
  unchanged.

### Tests
Added to `MapMatcherTest`: a single-point lateral spur dropped entirely; a
single-point connector dropped by the anchor floor and restored as a bridge
via `repairGaps`; a fork where the high-count branch is kept and a low-count
branch dropped; an out-and-back anti-parallel pair to a dead-end node
collapsing to a degree-1 leaf and being peeled. All existing
`MapMatcherTest`/`IntervalClusteringServiceTest` cases still pass.
`compileDebugKotlin` and `testDebugUnitTest` both pass.

## Issue #61 — Map-matching: perpendicular-distance snapping + direction-aware bearing gate
Completed: 2026-06-12 | Closed: pending

### What was done

**Perpendicular-distance snapping (`RTreeSpatialIndex.kt`)**
- `rebuildIndex` now decodes each edge's polyline once (falling back to the
  straight fromNode-toNode line when `geometryEncoded` is empty/short) and
  caches it in `edgeGeometries`, indexed by edge index/key.
- `queryEdgesNear` ranks candidates by true minimum point-to-segment
  (perpendicular) distance to the cached polyline via the new
  `nearestSegment`, instead of distance to the edge's bbox center.
- `EdgeCandidate` gained a `bearingDeg` field: the bearing of the polyline
  segment nearest the query point (computed in the same `nearestSegment`
  pass).
- `createBoundingBox` now takes the decoded geometry directly (no longer
  needs the edge/nodes map at query time).

**Direction-aware bearing gate (`MapMatcher.kt`)**
- New `computeHeadings(points)`: GPS heading per point as the bearing across
  a +/-2 point window (damps 1Hz jitter); returns null if the window
  collapses to a single coordinate.
- New `selectSnapCandidate(candidates, heading)`: from distance-sorted
  candidates, picks the nearest one within
  `INTERVAL_SNAP_BEARING_MAX_DIFF_DEG` (45deg) of `heading`; falls back to
  the nearest candidate overall if `heading` is null or none qualify (a
  point is never dropped purely due to heading noise). Both helpers are
  `internal` for direct unit testing.
- `snapPoints` now combines per-point candidates from `queryEdgesNear` with
  `computeHeadings` via `selectSnapCandidate`.
- `GeoUtils` gained `pointToSegmentDistance` (local equirectangular
  point-to-segment projection, meters) and `bearingDifference` (circular
  angle difference, 0-180).

### Tests
- New `RTreeSpatialIndexTest`: a point lying on a long edge ranks above a
  shorter edge whose bbox-center is closer; candidate `bearingDeg` matches
  the nearest segment's direction.
- New `MapMatcherTest` cases: a perpendicular dead-end spur near a junction
  is rejected in favor of the along-travel edge even though it's
  geometrically nearer; `selectSnapCandidate` falls back to nearest when no
  candidate matches heading and otherwise prefers the heading-matching
  candidate over a closer mismatch; `computeHeadings` uses a multi-point
  window.
- All existing `MapMatcherTest`/`IntervalClusteringServiceTest` cases still
  pass. `compileDebugKotlin` and `testDebugUnitTest` both pass.

## Issue #23 — Surface 'Reconnect Dropbox' prompt on persistent auth failure
Completed: 2026-06-11 | Closed: pending

### What was done

Distinguishes transient sync failures from persistent Dropbox auth failures
(revoked/invalid refresh token), since retrying the latter can never succeed
without the user reconnecting.

- `data/dropbox/DropboxSyncService.kt`: `sync()` now returns a new sealed
  `DropboxSyncResult` (`Completed`, `TransientFailure`, `NeedsReauth`) instead
  of a bare `List<ImportResult>`. The single `catch (e: DbxException)` around
  the list/download loop is classified via the new internal
  `isPersistentAuthFailure()`: `InvalidAccessTokenException` or a
  `DbxOAuthException` with `DbxOAuthError.INVALID_GRANT` (revoked refresh
  token) → `NeedsReauth`; everything else (network, rate limiting, server
  errors) → `TransientFailure` if no results were produced yet, otherwise the
  partial `Completed(results)` is still returned. The cursor is only saved for
  pages that completed successfully, so transient/auth failures leave sync
  state untouched for the next attempt. `downloadAndImport()` now rethrows
  persistent-auth `DbxException`s (instead of swallowing them as a per-file
  `ImportResult.Error`) so a token revoked mid-sync is still classified
  correctly by the outer catch.
- `data/dropbox/DropboxCredentialStore.kt`: added `setNeedsReauth()` /
  `needsReauth()` backed by the same `EncryptedSharedPreferences`, cleared
  together by the existing `clear()` on disconnect.
- `data/dropbox/DropboxAuthRepository.kt`: added `needsReauth: StateFlow<Boolean>`
  (seeded from the credential store) and `markNeedsReauth()`. Cleared on a
  successful `handleAuthResult()` (reconnect) and on `disconnect()`.
- `ui/screens/home/HomeViewModel.kt`: `syncDropbox()` now switches on
  `DropboxSyncResult` — `Completed` builds the existing snackbar message and
  triggers re-clustering as before, `TransientFailure` does nothing (silent
  retry next sync), `NeedsReauth` calls `markNeedsReauth()`. Also skips
  syncing entirely while already in the "needs reconnect" state, to avoid
  repeatedly hitting the failing token-refresh endpoint.
- `ui/screens/settings/SettingsViewModel.kt`: exposed
  `needsDropboxReauth = dropboxAuthRepository.needsReauth`.
- `ui/screens/settings/SettingsScreen.kt`: when connected and
  `needsDropboxReauth` is true, shows a warning row ("Dropbox sync needs
  reauthorization") plus a "Reconnect Dropbox" button above the existing
  "Disconnect" button. Tapping it reuses `connectDropbox()` (the same OAuth
  flow from #20); the prompt clears once `handleAuthResult()` succeeds.

### Tests
Added `DropboxSyncServiceTest` covering `isPersistentAuthFailure()` for
`InvalidAccessTokenException`, `DbxOAuthException` (`INVALID_GRANT` vs.
`INVALID_REQUEST`), `NetworkIOException`, `RetryException`, and
`ServerException`. `compileDebugKotlin` and `testDebugUnitTest` both pass.
