## Issue #18 — FastWayHomeService: time-optimized A* pathfinding to home
Completed: 2026-06-07 | Closed: 2026-06-07

### What was done

**`FastWayHomeService.kt` (new, `domain/service/`)**
- `@Singleton` service injecting `MapGraphRepository`, `EdgeStatsEstimator` (#17), and
  `UserSettingsRepository` (for `homeLat`/`homeLon`)
- `findFastWayHome(start: LatLng): FastWayHomeResult?` — locates the nearest graph edge to
  `start`, runs `aStarToHome()` toward the configured home location, converts the resulting
  edge sequence into `ScoredSegment`s, and scores it via `RideEstimator.estimateRide()` (#16)
  once per percentile band
- `FastWayHomeResult(path, slow, avg, fast, anySegmentsEstimated)` aggregates the three
  `RideEstimate`s plus whether any segment along the path used estimated (non-real) stats
- `aStarToHome()` is a brand-new, independent A* — `RoutePlanner.aStarReturn()` is untouched.
  Skeleton (priority queue `Node(idx, gCost, fCost)`, `gCosts`/`cameFrom`/`closedSet`,
  goal-radius check, path reconstruction) mirrors `aStarReturn`, but:
  - cost is **time-based**: `lengthM / speedP50` (converted to m/s), not distance-based
  - heuristic converts remaining haversine distance to a lower-bound time using
    `CyclingConstants.GPS_IMPLIED_MAX_SPEED_KMH` (120 km/h), keeping it admissible
  - target is the configured home location, not the route's start point
- Connectivity cache (`edgeByIndex`/`adjacency`/`edgeStartPoints`/`edgeEndPoints`) is built
  the same way as `RoutePlanner`'s (decode `geometryEncoded` via `PolylineDecoder`, fall back
  to node lat/lon, adjacency from `fromNode`/`toNode` topology), plus a precomputed
  `edgeStatsByIndex: Map<Int, EdgeStats>`: real stats for traversed edges with percentile
  data, `EdgeStatsEstimator.estimateStatsNear()` (with its 15 km/h fallback) for
  untraversed/data-sparse edges — precomputed once so the A* loop stays synchronous and fast
- No edges are excluded from the search graph; all are routable via the fallback-cost estimate

**`FastWayHomeServiceTest.kt` (new)**
- Small fixture chain graph (`N0 -> N1 -> N2 -> N3`/home, 2 traversed + 1 untraversed edge)
  confirms a concrete path is found end-to-end, `anySegmentsEstimated` is true (the untraversed
  edge falls back to 15 km/h since no nearby traversed edge shares its bearing), and
  Slow >= Avg >= Fast ride durations
- A disconnected two-component fixture confirms `findFastWayHome` returns `null` gracefully
- Added `io.mockk:mockk` as a test dependency to fake `UserSettingsRepository` (a concrete,
  `Context`-backed `DataStore` wrapper with no existing test-double pattern in the codebase)

### Tests
132 unit tests pass (the 4 pre-existing `FormatUtilsTest` locale-formatting failures are
unrelated — they fail identically on a clean `main`).

## Issue #17 — Nearest-traversed-edge statistic estimation primitive
Completed: 2026-06-07 | Closed: 2026-06-07

### What was done

**`domain/service/EdgeStatsEstimator.kt`**
- Added `EdgeStats` (per-percentile speed + nullable per-percentile power + `isEstimated`) and
  `EdgeStatsEstimator.estimateStatsNear(point: LatLng, bearing: Double): EdgeStats`
- Maintains its own private `RTreeSpatialIndex` instance built lazily (mutex-guarded) from only
  the *traversed* edges, so candidate `edgeKey`s map directly back to that traversed-only list —
  this sidesteps the index-key ambiguity that would arise from sharing `RoutePlanner`'s index
  (which is built over *all* edges and rebuilt/invalidated on its own schedule)
- Candidate selection: queries the spatial index within `EDGE_STATS_SEARCH_RADIUS_M` (50 m),
  filters out edges whose bearing differs from the query bearing by more than
  `EDGE_STATS_MAX_BEARING_DIFF_DEG` (90°, so roughly-opposite/parallel roads don't contribute),
  then takes the nearest `EDGE_STATS_NEAREST_K` (3)
- Distance-weights each candidate's percentile stats by `1 / (distanceM + 1)` (closer edges
  weigh more); power percentiles are skipped per-candidate when null and the overall power
  estimate is `null` if no candidate contributed
- Falls back to `EDGE_STATS_FALLBACK_SPEED_KMH` (15 km/h) with `null` power when no traversed
  edge is found within range; `isEstimated` is always `true` (this function only fills gaps —
  edges with real stats aren't routed through it)
- Added `EDGE_STATS_NEAREST_K`, `EDGE_STATS_SEARCH_RADIUS_M`, `EDGE_STATS_FALLBACK_SPEED_KMH`,
  `EDGE_STATS_MAX_BEARING_DIFF_DEG` to `CyclingConstants`

### Deviation from the issue's literal `EdgeStats` signature
- The issue sketch shows non-null `Double` power percentiles; kept them nullable (`Double?`)
  to mirror `MapEdge`'s own nullable power stats and `RideEstimator.ScoredSegment`'s nullable
  power fields (#16) — `RideEstimator.estimateRide` already skips segments with no power data,
  so this is the natural fit for the eventual #18 conversion to `ScoredSegment`

### Tests
Added `EdgeStatsEstimatorTest` (3 cases): distance-weighted averaging favors the nearer of two
same-direction traversed edges, fallback to the 15 km/h conservative estimate when nothing is
within the search radius, and bearing filtering excludes a closer-but-opposite-direction edge
from the average. 124/128 unit tests pass (the 4 failures are the pre-existing `FormatUtilsTest`
locale-formatting issues — `,` vs `.` decimal separator — noted in #16, unrelated to this change).

## Issue #16 — RideEstimator: weighted speed/power/duration scoring module
Completed: 2026-06-07 | Closed: 2026-06-07

### What was done

**`domain/service/RideEstimator.kt`**
- Added graph-agnostic `ScoredSegment` (lengthM + per-percentile speed/power + `isEstimated`),
  `Percentile` enum (SLOW/AVG/FAST → 25th/50th/75th), and `RideEstimate` result type — none
  depend on `MapEdge`, the graph repository, or UI types, so the module can be reused both for
  the "fast way home" estimate (#9/#18) and for future `.gpx` rating
- `RideEstimator.estimateRide(segments, percentile)`:
  - Average speed: each segment's chosen-percentile speed weighted by `lengthM`, summed and
    divided by total length; converted to overall duration via `(totalKm / avgSpeedKmh) * 3600`
  - Average power: each segment's own duration (`lengthM / its percentile speed`, in seconds)
    used as the weight for that segment's percentile power; weighted sum divided by the total
    duration of only the segments that have power data for that percentile (others skipped)
  - `anyEstimatedSegments` is `true` if any input segment has `isEstimated = true`
  - Empty input returns a zeroed `RideEstimate` rather than dividing by zero

### Pre-existing issue fixed along the way
- `FakeCyclingSessionRepository` (test fixture) was missing the `getAllClusterData` /
  `getSessionsByIdsList` overrides added to `CyclingSessionRepository` during issue #2,
  which broke compilation of the entire unit test source set

### Tests
Added `RideEstimatorTest` (6 cases): hand-computed duration/power for Slow/Avg/Fast on a
two-segment fixture, zero-segment input, segments with no power data excluded from the power
average, and the `anyEstimatedSegments` flag. 121/125 unit tests pass (the 4 failures are
pre-existing `FormatUtilsTest` locale-formatting issues — `,` vs `.` decimal separator — unrelated
to this change).

## Issue #1 — Repeated Routes: too few clusters + perceived recalculation on tab open
Completed: 2026-06-05 | Closed: 2026-06-05

### What was done

**Clustering algorithm (`domain/service/RouteClusteringService.kt`)**
- Replaced complete-linkage agglomerative clustering with **single-linkage / connected
  components** (BFS over the qualifying-pair graph). Complete-linkage required every pair
  in a cluster to independently meet 85% similarity, which shattered clusters on the first
  GPS-noise outlier
- Relaxed length-delta filter to accommodate ±10% per-ride length variation:
  `LENGTH_DELTA_FRACTION` 0.10 → 0.15, `MAX_LENGTH_DELTA_KM` 5 → 7
- Kept `SIMILARITY_THRESHOLD` (0.85), `MATCH_THRESHOLD_M` (25), `MIN_GROUP_SIZE` (3) unchanged
- Deleted `runIncrementalClustering` (~95 LOC); single code path now — full clustering runs
  after each import (in background) and on pull-to-refresh
- Added `DEBUG_LOGGING` compile-time const: when enabled, logs per-pair accept/reject reasons
  with actual length-Δ / centroid-Δ / similarity values, plus a summary
  `N sessions → M edges → K components → L ≥ MIN_GROUP_SIZE`

**Background execution + caching for Routes tab**
- Created `di/CoroutineScopeModule.kt`: `@ApplicationScope` qualifier + `@Singleton` provider
  for `CoroutineScope(SupervisorJob + Dispatchers.Default)`
- Created `data/cache/RepeatedRoutesCache.kt` (`@Singleton`) hoisting
  `repository.getAllRoutes()` into a hot `StateFlow` collected on the app scope; survives
  ViewModel recreation, so tab switches no longer trigger Room re-query + GPS track JSON
  re-parsing, and the loading spinner only ever shows on cold app start
- `RepeatedRoutesViewModel`: now reads `routes` and `isLoading` from the cache instead of
  collecting the repository Flow directly
- `HomeViewModel.importFromUris`: per-file `runIncrementalClustering` call deleted; after
  the batch loop completes, `appScope.launch { runClustering() }` re-clusters on a scope
  that survives the user navigating away from the dashboard

### Known accepted limitation
If the new algorithm splits an existing custom-named cluster into multiple, the name only
carries to a child that's an exact subset of the old cluster (existing name-preservation
logic in `RouteClusteringService.kt`). Newly-split routes may need to be re-named once.

### Tests
Pre-existing unit tests pass; `compileDebugKotlin` clean (only pre-existing warnings remain).
