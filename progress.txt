## Issue #25 — RepeatedInterval: clustering, persistence, and management screen
Completed: 2026-06-08 | Closed: 2026-06-08

### What was done

**`RepeatedInterval` domain model + persistence**
- New `domain/model/RepeatedInterval` (`id`, `name`, `intervals: List<IntervalSession>`,
  `edges: List<MapEdge>`, `startLat/startLon/endLat/endLon`, `distanceM`)
- New `repeated_intervals` Room table (`RepeatedIntervalEntity`/`RepeatedIntervalDao`,
  `MIGRATION_6_7`, DB version 6→7) storing `intervalIds` and `edges` as JSON — the latter as
  `[fromNode, toNode]` reference pairs resolved against `MapGraphRepository.getAllEdges()` at
  read time (mirrors how `RepeatedRouteEntity.sessionIds` resolves against `CyclingSessionRepository`)
- `RepeatedIntervalRepository`/`RepeatedIntervalRepositoryImpl` mirroring
  `RepeatedRouteRepository`'s flow-combining read paths and rename/delete/save operations;
  bound in `RepositoryModule` and `DatabaseModule`
- `RepeatedIntervalsCache` (singleton hot `StateFlow`) mirroring `RepeatedRoutesCache`

**`IntervalClusteringService` (`domain/service/IntervalClusteringService.kt`)**
- Adapts `RouteClusteringService`'s single-linkage/connected-components structure to raw
  `IntervalSession`s: pairs qualify when length delta ≤ `INTERVAL_LENGTH_TOLERANCE_M` (100 m)
  AND the bidirectional max GPS-point coverage (neighbor within `INTERVAL_POINT_MATCH_RADIUS_M`
  = 20 m) is ≥ `INTERVAL_POINT_SIMILARITY_THRESHOLD` (0.8) — using its own `SpatialGrid`
  (same cell-bucket pattern as `RouteClusteringService`'s)
- No minimum group size — every distinct shape becomes an archetype, even singletons
- Picks each cluster's median-`distanceM` interval, map-matches its track via `MapMatcher` for
  `List<MapEdge>` geometry, and derives `distanceM`/start-end lat-lon by decoding the first/last
  edge's `geometryEncoded` via `PolylineDecoder`; clusters whose representative can't be
  map-matched are dropped
- **Subset discarding**: processes archetypes longest-first, discarding any whose edges overlap
  an already-kept (longer) archetype's edges by ≥ `INTERVAL_SUBSET_OVERLAP_THRESHOLD` (0.8) of
  its own `distanceM`
- Preserves existing names/IDs across re-runs by matching new clusters to existing entries via
  overlapping raw-interval-ID subsets (same precedent as `RouteClusteringService.kt:171-186`)

**New named constants** (`util/Constants.kt`): `INTERVAL_LENGTH_TOLERANCE_M = 100.0`,
`INTERVAL_POINT_SIMILARITY_THRESHOLD = 0.8`, `INTERVAL_POINT_MATCH_RADIUS_M = 20.0`,
`INTERVAL_SUBSET_OVERLAP_THRESHOLD = 0.8`

**UI**
- `RepeatedIntervalsScreen`/`-ViewModel` (list, sort, manual pull-to-refresh → `runClustering()`)
  and `RepeatedIntervalDetailScreen`/`-ViewModel` (stats, matched-edge map preview via
  `PolylineDecoder` + `MapTrackRenderer`, occurrence list, rename) mirroring
  `RepeatedRoutesScreen`/`RepeatedRouteDetailScreen`
- `RepeatedRoutesScreen` now hosts a `SingleChoiceSegmentedButtonRow` ("Routes"/"Intervals")
  switching between the existing route list and the new interval list, each backed by its own
  ViewModel/cache; sort-menu options switch with the selected tab
- Wired `repeated_interval/{intervalId}` into `Screen`/`VelometricsNavHost`

### Note on `IntervalPrototypeRoute` removal
Left `IntervalPrototypeRoute`/`IntervalMatcher`/`IntervalGroup` in place — they're still actively
wired into `FitImportService`/`MapViewViewModel`, and rewriting `IntervalMatcher` to assign
against `RepeatedInterval` archetypes (instead of removing its dependency out from under it) is
explicitly issue #26's job ("blocked by #25"). Removing them here would have meant doing #26's
work prematurely and leaving it half-finished.

## Issue #24 — Map-matching: snap GPS tracks to coherent road-graph edge sequences
Completed: 2026-06-08 | Closed: 2026-06-08

### What was done

**`MapMatcher` service (`domain/service/MapMatcher.kt`)**
- New `@Singleton` service taking a raw GPS track (`List<List<Double>>`, the decoded shape of
  `IntervalSession.gpsTrack`) and returning a connected, ordered `List<MapEdge>`, or `null` if
  no coherent match could be produced
- Lazily builds a node-adjacency map (`fromNode`→`toNode` successor lookup, the same
  edge-indexing/successor-map pattern as `FastWayHomeService.buildConnectivity`) and an
  `RTreeSpatialIndex` over all graph edges (the same index `EdgeStatsEstimator` uses), guarded
  by a `Mutex` and cached across calls
- **Snapping**: each GPS point is snapped to the nearest edge via `spatialIndex.queryEdgesNear`
  within the new `INTERVAL_EDGE_SNAP_RADIUS_M` constant
- **Outlier removal**: consecutive identical snaps are collapsed into runs; a single-point run
  sandwiched between two runs that are themselves mutually adjacent is dropped as GPS noise
  (bounded by the new `INTERVAL_MATCH_MAX_OUTLIER_RUN_POINTS` constant)
- **Gap repair**: walks the deduped snapped sequence; wherever consecutive edges aren't
  graph-adjacent, runs a bounded BFS (`findConnectingPath`, capped by the new
  `INTERVAL_MATCH_MAX_REPAIR_DEPTH` constant) over the local adjacency graph and splices the
  connecting edges in
- **Rejection**: if a gap can't be bridged within the max repair depth (e.g. the track jumps
  between disconnected graph components), `matchTrack` returns `null` rather than producing an
  incoherent multi-segment result

**New named constants** (`util/Constants.kt`): `INTERVAL_EDGE_SNAP_RADIUS_M = 30.0`,
`INTERVAL_MATCH_MAX_REPAIR_DEPTH = 6`, `INTERVAL_MATCH_MAX_OUTLIER_RUN_POINTS = 1`

**Fixed pre-existing test breakage**
- `NavigationViewModelTest.buildViewModel()` was missing the `fastWayHomeService` and
  `userSettingsRepository` constructor args added to `NavigationViewModel` in #18/#19 — added
  `mockk` fakes for both (this was blocking compilation of the entire unit-test source set)

**Tests** — `MapMatcherTest` covers the three acceptance-criteria cases: a clean track snapping
to a fully connected sequence, a noisy track with a missed middle edge repaired via local
adjacency search, and a track jumping between disconnected graph components rejected with `null`

### Tests
133 unit tests pass.

## Issue #26 — Replace prototype-matching with RepeatedInterval-based assignment
Completed: 2026-06-08 | Closed: 2026-06-08

### What was done

**`IntervalMatcher` rewrite (`domain/service/IntervalMatcher.kt`)**
- Replaced prototype-route matching (50m start/end proximity against `IntervalPrototypeRoute`)
  with `RepeatedInterval`-archetype matching using the same length + GPS-point-overlap
  similarity test the clustering step uses (length delta ≤ `INTERVAL_LENGTH_TOLERANCE_M`,
  ≥ `INTERVAL_POINT_SIMILARITY_THRESHOLD` of points within `INTERVAL_POINT_MATCH_RADIUS_M`)
- Extracted that shared similarity test into a new `IntervalSimilarity` object (`PreparedTrack`,
  `SpatialGrid`, `qualifies`) so `IntervalClusteringService` and `IntervalMatcher` use one
  consistent definition of "matches" — `IntervalClusteringService` now delegates to it
- An archetype's comparison track is built by decoding+concatenating its matched road-graph
  edges' polylines (`PolylineDecoder.decode(edge.geometryEncoded)`)
- When a raw interval qualifies against multiple archetypes, it's assigned to the one with the
  greatest `distanceM` ("the longer one", per the issue's tie-break rule)
- `matchToRepeatedIntervals` persists assignments by appending matched intervals directly onto
  `RepeatedInterval.intervals` and saving via `RepeatedIntervalRepository` — `intervalIds` on
  the entity remains the single source of truth (no redundant per-interval FK column)

**Removed `prototypeRouteId` and `IntervalPrototypeRoute`/`IntervalGroup`**
- Dropped `IntervalSession.prototypeRouteId`; Room migration `MIGRATION_7_8` (DB v7→v8) recreates
  `interval_sessions` without the column/FK and drops `interval_prototype_routes`
- Deleted `IntervalPrototypeRoute`, `IntervalPrototypeRouteEntity`, `IntervalPrototypeRouteDao`
  and all repository/mapper/DI references
- Removed `IntervalGroup`; `MapOverlayUtils.groupIntervals` now partitions raw intervals against
  `RepeatedInterval` archetypes directly (an archetype "groups" iff `intervals.isNotEmpty()`),
  with `avgDurationNormalizedSec`/`avgPower` as plain functions over `RepeatedInterval`
- Updated all call sites: `MapIntervalRenderer` (renders `RepeatedInterval` groups, GeoJSON
  property renamed `prototypeId` → `repeatedIntervalId`), `MapViewViewModel`
  (`allRepeatedIntervals` replaces `allPrototypeRoutes`), `MapViewScreen` (click-handler lookup
  by `RepeatedInterval.id`, `PrototypeGroupSheet` now takes a `RepeatedInterval`)
- `IntervalRepository.insertIntervals` now returns the inserted `List<Long>` so
  `FitImportService` can pass DB-assigned ids into the matcher (raw `IntervalDetector` output
  has `id = 0`, but the matcher needs real ids to record into `RepeatedIntervalEntity.intervalIds`)

**Tests** — rewrote `IntervalMatcherTest` (unambiguous match, no match, multi-match
tie-break-by-longer-archetype), `MapOverlayUtilsIntervalTest` (grouping/averaging against
`RepeatedInterval`), and `MapViewViewModelTest` (`FakeRepeatedIntervalRepository` replaces the
removed prototype-route fakes)

### Tests
All unit tests pass.

## Issue #34 — Create Activity-scoped GpxSharedViewModel as GPX state foundation
Completed: 2026-06-09 | Closed: pending

### What was done

**`GpxSharedViewModel` (`ui/shared/GpxSharedViewModel.kt`)**
- New `@HiltViewModel` scoped to the Activity (injectable via `hiltViewModel()` with the
  Activity's `ViewModelStoreOwner`) so GPX state survives tab switching within a session but
  does not persist across app kills
- Exposes `gpxTrack: StateFlow<GpxTrack?>`, `gpxPois: StateFlow<List<PoiWithDistances>>`,
  and `isLoadingPois: StateFlow<Boolean>`
- `loadGpxFromUri(uri, contentResolver)` — parses the GPX on `Dispatchers.IO` via `GpxParser`,
  sets `_gpxTrack`, then immediately calls `fetchPoisForTrack()`
- `clearGpx()` — resets both `_gpxTrack` and `_gpxPois` to null/empty
- `fetchPoisForTrack(track)` — private; uses the same corridor-filter (200 m) and projection
  logic as `NavigationViewModel.doFetchAllTrackPois()`: queries `MapGraphRepository.getAllPois()`,
  filters each POI by perpendicular distance to track, projects the remainder onto the track
  geometry, and sorts by `trackDistanceM` from track start. Computes distances from the track's
  first point (eager load, no GPS fix required).
- No UI changes — purely ViewModel/domain layer per the issue scope.
